## Context

`main()` 在 `PULL_ALL` 解析為空時，會落到 else 分支掃描父目錄下所有 git repo（`index.js:374-378`）。`env()` 的 `'' || null` 讓「`.env` 沒這行」與「`PULL_ALL=` 空值」回傳相同結果，因此「沒設定」與「想掃全部」本就無法區分。`runInit()` 又把「全不勾」寫成 `PULL_ALL=`（空），顯示「（全部）」（`index.js:471-475`）——三者共同構成「空 = 全部」的隱性契約。本變更移除此契約。

## Goals / Non-Goals

**Goals:**
- `PULL_ALL` 未設定時停下並引導 `pull-all init`，不掃描任何 repo。
- `init` 全不勾視為取消、不寫入，避免「沒設定 → 引導 init → 全不勾寫空 → 又被當沒設定」的迴圈。

**Non-Goals:**
- 不新增「明確掃全部」的特殊值（如 `PULL_ALL=*`）。掃全部能力整個退場，未來如有需要另開提案。
- 不修改 `env()` 對 null 與空字串的處理；既然兩者殊途同歸（都引導 init），無須區分。
- 不改 `clone` 的核心邏輯，只統一其未設定時的訊息措辭。

## Decisions

**移除掃全部，而非保留為特殊值** — 使用者實際痛點是「沒設定卻被靜默拉去掃一堆」。保留 `PULL_ALL=*` 之類入口會重新引入「空 vs 全部」的區分負擔，且 `init` 已能快速建立白名單，掃全部的便利性不值得這份複雜度。

**`init` 全不勾 = 取消不寫入，而非強制至少選一個** — 取消較貼近 CLI 慣例：保留既有 `.env`、不打斷。第一次跑且無既有值時全不勾 → 下次 `main` 仍引導 `init`，這是合理的「必須做選擇」而非迴圈 bug。

## Risks / Trade-offs

- [既有使用者依賴「不設定就掃全部」] → 升級後 `main` 會引導跑一次 `pull-all init`；屬一次性、低成本，且訊息明確指出下一步。標為 **BREAKING** 並於 proposal/README 載明。
- [第一次使用者全不勾感覺「沒效果」] → 取消訊息明確說「未選任何 repo，已取消」，使用者知道需重跑並勾選。

## Migration Plan

1. 既有依賴掃全部的使用者執行 `pull-all init` 勾選要追蹤的 repo（一次性）。
2. 無 rollback 顧慮：純行為與訊息變更，不涉及資料或結構。
